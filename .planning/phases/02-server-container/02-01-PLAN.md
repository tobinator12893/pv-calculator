<task type="auto">
  <name>Task 2: Write multi-stage server Dockerfile</name>
  <files>pv-calculator/server/Dockerfile</files>
  <action>
Create pv-calculator/server/Dockerfile with two stages:

**Stage 1 — builder:**
- `FROM node:24-bookworm-slim AS builder`
- `WORKDIR /app`
- Copy package.json and package-lock.json first (layer cache optimization per BUILD-04 pattern)
- `RUN npm ci` (full install — devDependencies needed for tsc)
- Copy tsconfig.json, then copy src directory
- `RUN npm run build` (runs tsc, outputs to ./dist)

**Stage 2 — runtime:**
- `FROM node:24-bookworm-slim AS runtime`
- `WORKDIR /app`
- `ENV NODE_ENV=production` (MUST be in runtime stage only, NOT builder — npm ci in builder needs devDeps)
- Copy package.json and package-lock.json
- `RUN npm ci --omit=dev` (production deps only)
- `COPY --from=builder --chown=node:node /app/dist ./dist` (chown is CRITICAL — without it files are root-owned and node user may not access them)
- `USER node` (built-in node user UID 1000 — do NOT use RUN adduser)
- `EXPOSE 3001`
- `CMD ["node", "dist/index.js"]` (direct node invocation, NOT npm start — ensures SIGTERM reaches Node.js)

Add a syntax directive as first line: `# syntax=docker/dockerfile:1`
Add stage separator comments: `# --- Stage 1: Build ---` and `# --- Stage 2: Runtime ---`

Things to AVOID (from research pitfalls):
- Do NOT set NODE_ENV=production in builder stage (breaks npm ci for devDeps)
- Do NOT use `COPY . .` (breaks layer caching)
- Do NOT use `CMD ["npm", "start"]` (npm intercepts SIGTERM)
- Do NOT use `RUN adduser` (node user already exists in base image)
- Do NOT omit --chown=node:node on COPY --from=builder
  </action>
